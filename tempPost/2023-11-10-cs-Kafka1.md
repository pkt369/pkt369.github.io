# Apache Kafka 란

카프카란 데이터를 스트림 파이프라인을 통해 실시간으로 관리하기 위한 분산 스트리밍 플랫폼입니다.



## 기본 구성 요소

- Cluster: 여러대의 컴퓨터(서버)들이 연결되어 하나의 시스템처럼 동작하는 컴퓨터의 집합
- Producer: 데이터를 제공하는 전달자 (제공자)
- Consumer: 데이터를 브로커를 통해 받아 소비하는 소비자
- Broker: 생성자와 소비자의 사이를 중재하는 중재자
- Topic: 데이터를 구분하는 역할 (카테고리)
- Partition: 토픽을 구성하는 데이터 저장소, 수평확장 가능 (Queue)



## Kafka  흐름

- Producer 가 Topic 이 적힌 메세지를 Broker 에게 전달해줍니다.
- Broker 는 Topic 을 보고 해당하는 Partition 에 저장합니다.
- Consuemr 가 카프카에 접근하여 원하는 Topic 을 달라고 하면 Broker 가 해당하는 Partition 에 접근하여 데이터를 가져와 전달해줍니다.

(그림)



#### Details

- Broker 가 Partition 으로 전달할 때 Topic 을 기준으로 전달을 해줍니다. 따라서 하나의 Topic 당 한개 이상의 Partition 이 존재합니다.
- Partition 갯수는 상관없으며, 늘릴수만 있고 줄일 순 없습니다.
- 파티션 내부에 각 offset (고유 번호) 로 관리하고 있으며, Consumer 에게 제공 후 offset 이 하나 늘어나는 방식으로 동작합니다.
- 파티션이 여러개면 라운드 로빈 방식으로 동작하고, 파티션 갯수가 많을수록 처리량은 좋아지나 에러 복구하는 시간이 늘어납니다.
- Consumer 가 메세지를 처리해도 메세지는 삭제되지 않고, 하나의 메세지를 여러번 처리할 수 도 있습니다.
- Topic 을 관리하는 중앙본부 역할은 Zookeeper 가 합니다.



## 궁금증 해결하기

### 왜 여러개의 Partition 으로 설계하나?

우리가 데이터베이스에 접근할 때 쓰고 있을 때 락이 걸린 상태이고 락이 풀려야 접근할 수 있는 것처럼 Partition 도 같은 방식으로 동작합니다.

Broker 가 만약 몇천건의 데이터 한번에 작성된다고 가정했을 때 Consumer 는 병목현상으로 데이터를 바로 받지못할 수 있습니다.

이러한 현상을 해결하기 위해 여러개의 Partition 으로 사용합니다. 하지만 위에서 언급했듯이 늘린다음 줄일 수 있기때문에 충분히 검토된 다음에 늘려야 합니다.



### 에러 발생시 카프카는 어떻게 동작하나

운영을 하다보면 의도치 않게 에러가 발생하는 경우가 있습니다. 



## 다른 종류의 Message Queue

하지만 Kafka 만 메세지 큐 방식으로 동작하는 건 아닙니다. 같은 방식으로는 Rabbit MQ, Redis 등이 존재합니다. 모두 메세지 큐 방식을 사용하는 건 같으나 Broker 의 유형에 따라 동작방식이 달라집니다.



### Broker

#### 메세지 브로커

- 보통 비동기로 사용하기 위해 채택하며, 위에서 설명한 것과 같이 배달원 역할을 합니다. 이러한 구조를 보통 pub/sub 구조라고 말하며, 대표적으로 Rabbit MQ, Redis 가 있습니다.

#### 이벤트 브로커

- 메세지 브로커의 역할을 그대로 수행하며, 다른점으로는 Producer 가 생성한 이벤트를 이벤트 처리 후 바로 삭제하지 않고 저장합니다.  예를 들어 에러가 발생했을 때 그 시점부터 다시 이벤트를 처리할 수 있습니다.
- 또한 메세지 브로커보다 더 많은 양을 처리할 수 있습니다.
- 대표적으로 Kafka 가 있습니다.
